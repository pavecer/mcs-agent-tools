# o3 — System Instruction Best Practices

o3 is OpenAI's most advanced reasoning model, capable of PhD-level problem solving across science, coding, and analysis. Like o1, it performs deep internal chain-of-thought reasoning — instructions should focus entirely on goals, constraints, and output requirements rather than process steps.

---

## Key Principle: Leverage Reasoning — Don't Direct It

o3 excels when given clear goals, precise constraints, and a well-defined output structure. Avoid directing its internal reasoning.

| ❌ Avoid | ✅ Use Instead |
|---|---|
| "Analyse this step by step" | "Provide a complete analysis covering A, B, and C" |
| "First identify X, then reason about Y" | "Output must address X and Y with evidence from sources" |
| "Think carefully" | "Be thorough and cite all claims" |

---

## 1. Provide Precise Output Specifications

o3 can generate highly structured output reliably. Define schemas for complex tasks.

```
Summarise each legal clause as follows:
- Clause title: [name]
- Applies to: [scope]
- Key obligation: [1 sentence]
- Risk if breached: [low | medium | high]
- Source: [document name and section]
```

---

## 2. Strong Grounding Rules

o3's extensive training knowledge makes grounding constraints critical for factual accuracy on private or proprietary data.

```
## Grounding Rules
Answer exclusively from the documents and search results provided in each query.
Your responses must be traceable to a specific source.
Do not use general knowledge not present in the grounding data.
If you cannot find an answer in the sources, explicitly state: "Not found in available sources."
```

---

## 3. Include Self-Validation Criteria

o3 can verify its own output against explicit criteria — leverage this capability.

```
Before responding, ensure that:
1. Every factual claim references a specific source document.
2. No information has been inferred beyond what is explicitly stated in the source.
3. The response format matches the required structure.
```

---

## 4. Persona and Authority Level

Be explicit about who the agent represents and the level of authority its answers carry.

```
You are an AI research assistant for the [Your Organisation] Strategy Team.
You provide analysis and summaries based on provided documents only.
You do not make final recommendations — you inform human decision-makers.
Every analysis must include a source citation.
```

---

## 5. Scope and Escalation

```
You operate exclusively within [defined domain].
Do not answer questions outside this domain; redirect the user to [contact / channel].
If a question cannot be answered from available sources, say so explicitly.
```

---

## 6. Avoid Redundant Reasoning Instructions

These are **not needed** with o3 and consume token budget:

- "Think step by step"
- "Reason through the problem"
- "Consider all perspectives"
- "Let's work through this carefully"

---

## 7. Recommended Lengths

| Use Case | Target Length |
|---|---|
| Focused expert assistant | 150–600 chars |
| Enterprise knowledge agent | 500–5,000 chars |
| Complex multi-constraint agent | 1,000–15,000 chars |

---

## 8. Common Mistakes

- **Not leveraging self-validation:** o3 can verify its own output — include explicit validation criteria as part of instructions.
- **Over-specifying reasoning:** Like o1, o3's internal reasoning is superior to prompted reasoning; avoid directing the process.
- **Missing output schema for structured tasks:** o3's precision is wasted without a defined output format.
- **Weak grounding for private data:** o3's broad training knowledge means grounding must be enforced more strictly than for simpler models.
